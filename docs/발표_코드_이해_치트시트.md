# BookLog 코드 전체 흐름 이해 도우미

이 문서의 목표는 코드를 외우는 게 아니라, **"어떤 화면이 어떤 파일에 있고, 데이터가 어디서 와서 어디에 저장되는지"** 를 머릿속에 그릴 수 있게 하는 것이다. 한 번 읽고 나면 아무 파일이나 열어도 "이게 전체 그림에서 어디쯤인지" 감이 잡히는 걸 목표로 한다.

> 발표 시연 멘트가 필요하면 `발표_기능별_코드_설명집.md`를 보면 된다. 이 문서는 순수하게 **흐름 이해용**이다.

---

## 0. 한 문장 요약

BookLog는 서버 없이 Android 앱 내부 SQLite DB에 회원·책·리뷰를 저장하고, 카카오 도서 검색 API로 외부 책 정보를 가져와 리뷰를 남길 수 있는 도서 리뷰 앱이다.

---

## 1. 전체 구조: 4개 층으로 보기

코드가 많아 보여도 역할은 4개 층뿐이다. 이 그림 하나만 기억하면 된다.

```text
┌─ 화면 (Activity) ─────────────────────────────────────┐
│  LoginActivity, SignupActivity, HomeActivity,         │
│  SearchActivity, BookDetailActivity,                  │
│  WriteReviewActivity, MyReviewActivity, MyPageActivity│
└───────────────────────────────────────────────────────┘
        │ 목록 데이터를 화면에 그릴 때
        ▼
┌─ 목록 어댑터 (Adapter) ───────────────────────────────┐
│  BookAdapter, ReviewAdapter, MyReviewAdapter          │
└───────────────────────────────────────────────────────┘
        │ 데이터를 읽고/쓸 때
        ▼
┌─ 데이터 저장/조회 (단일 출처) ────────────────────────┐
│  DBHelper (SQLite) — users / books / reviews 테이블   │
└───────────────────────────────────────────────────────┘

┌─ 보조 유틸 (어디서나 호출) ───────────────────────────┐
│  Session(로그인상태), KakaoApi(외부검색),             │
│  Covers·ImageLoader(표지), BottomNav(하단탭), Pw(해시)│
└───────────────────────────────────────────────────────┘
```

핵심 한 줄: **Activity는 화면, Adapter는 목록 그리기, DBHelper는 데이터, 나머지는 보조 도구.** 데이터의 출처는 언제나 `DBHelper` 하나다.

---

## 2. 데이터가 어디 사는가 — 테이블 3개

모든 데이터는 `DBHelper.onCreate()`에서 만드는 세 테이블에 들어간다. 이것만 알면 어떤 화면이든 "이 데이터는 어느 테이블에서 오는가"를 추적할 수 있다.

```text
users    : user_id, username, pw_hash(해시된 비번), nickname
books    : book_id, title, author, ... , cover, base_count, base_sum
reviews  : review_id, book_id, user_id, nickname, rating, content,
           is_spoiler, helpful_count, created_at
```

- `books.base_count / base_sum` = **미리 깔아둔 가짜 커뮤니티 평점**. 화면이 비어 보이지 않도록 시드 단계에서 넣어둔 값. (→ 9번 평균 별점에서 다시 등장)
- `reviews.user_id = 0` 인 행 = 시드된 커뮤니티 리뷰. `user_id = 내 id` 인 행 = 내가 쓴 리뷰.

## 3. 앱이 켜지는 흐름

관련 파일: `AndroidManifest.xml` · `LoginActivity.java` · `Session.java`

```text
앱 실행
  → Manifest에서 MAIN/LAUNCHER로 등록된 LoginActivity가 시작점
  → LoginActivity가 Session.isLoggedIn() 으로 로그인 상태 확인
  → 이미 로그인 → 바로 HomeActivity
  → 아니면 → 로그인 화면 표시
```

```java
if (Session.isLoggedIn(this)) {   // SharedPreferences에 user_id가 있나?
    goHome();
    return;
}
```

이해 포인트:
- `Activity` = 화면 하나. `Intent` = 화면 이동/데이터 전달 객체.
- `Session`은 DB가 아니라 **SharedPreferences**(앱 내부 간단 저장소)에 로그인 정보를 둔다. 그래서 앱을 껐다 켜도 로그인이 유지된다.

---

## 4. 로그인 / 회원가입 — 비번은 해시로

관련 파일: `LoginActivity` · `SignupActivity` · `DBHelper` · `Pw` · `Session`

```text
[로그인]  아이디/비번 입력 → DBHelper.login() → users 조회 성공 → Session.login() → 홈
[회원가입] 아이디/닉네임/비번 입력 → DBHelper.signup() → users에 저장 → 로그인 화면 복귀
```

이해 포인트:
- 비밀번호는 평문으로 저장하지 않는다. `Pw.hash()`로 **SHA-256 해시**한 뒤 저장하고, 로그인 때도 입력 비번을 같은 방식으로 해시해 **해시끼리 비교**한다. → DB가 새도 원래 비번을 알 수 없다.
- 로그인 성공 정보(id/username/nickname)는 `Session`이 들고 다닌다. 이후 모든 "내" 데이터(내 리뷰, 마이페이지)는 `Session.userId()`를 기준으로 조회된다.

## 5. 홈 화면과 인기 도서

관련 파일: `HomeActivity` · `DBHelper.getPopularBooks()` · `BookAdapter`

```text
HomeActivity
  → DBHelper.getPopularBooks() 로 책 목록 조회
  → BookAdapter가 ListView에 책 카드로 표시
```

"인기"의 기준이 별점이 아니라 **리뷰 수**라는 게 포인트다.

```text
total_count = base_count + 실제 reviews 수
정렬:  ORDER BY total_count DESC   (리뷰 많은 책이 위로)
```

## 6. 검색이 두 종류라는 것 — 가장 헷갈리는 부분

검색은 **로컬 DB 검색**과 **카카오 API 검색** 두 가지이고, 트리거가 다르다. 여기만 정확히 구분하면 검색 흐름은 끝이다.

```text
① 홈 검색창에 "글자를 입력하는 중"
     → TextWatcher → DBHelper.searchBooks(query)
     → 이미 DB에 있는 책 제목만 LIKE로 필터링 (네트워크 X)

② 키보드의 "검색 버튼"을 누름
     → SearchActivity로 이동 (검색어를 Intent로 전달)
     → KakaoApi.search(query) → 카카오 도서 API 호출
     → DB에 없던 외부 책까지 검색 (네트워크 O)
```

한 줄 정리: **입력 중 = 로컬 필터, 검색 실행 = 외부 API.**

---

## 7. 검색 결과를 누르면 — 저장 후 상세로

관련 파일: `SearchActivity` · `DBHelper.insertOrGetBook()` · `BookDetailActivity`

```text
카카오 검색 결과 클릭
  → DBHelper.insertOrGetBook()
       같은 제목+저자가 이미 있으면 → 기존 book_id 재사용
       없으면 → books 테이블에 새로 저장
  → book_id 를 Intent로 넘겨 BookDetailActivity 오픈
```

포인트: 카카오 결과는 아직 "우리 DB의 책"이 아니다. **상세로 들어가는 순간에만** DB에 저장하고, 이후로는 `book_id`만으로 다룬다. (중복 저장은 `insertOrGetBook`이 막아준다.)

---

## 8. 책 상세 화면 — book_id 하나로 재조회

관련 파일: `BookDetailActivity` · `DBHelper.getBook()` · `DBHelper.getReviewsForBook()` · `ReviewAdapter`

```text
BookDetailActivity
  → Intent에서 book_id 받음
  → DBHelper.getBook(bookId)            → 책 정보 + 평균 별점 (상단 헤더)
  → DBHelper.getReviewsForBook(bookId)  → 리뷰 목록
  → ReviewAdapter가 리뷰를 ListView에 표시
```

포인트: 상세 화면은 화면 자체에 책 정보를 들고 오지 않고 **book_id만 받아 DB에서 다시 조회**한다. 그래서 홈에서 들어오든, 검색 결과에서 들어오든, 내 리뷰에서 들어오든 **같은 상세 화면을 재사용**한다.

### 8.1 리뷰 정렬

```text
정렬 글자 클릭 → AlertDialog(최신순/별점순) → sortByRating 값 변경 → loadReviews() 재호출
```

리스트를 화면에서 직접 섞지 않는다. **SQL의 ORDER BY만 바꿔서 DB가 정렬한 결과를 다시 가져온다.** (별점순 = `rating DESC`, 최신순 = `created_at DESC`)

### 8.2 스포일러 리뷰 처리

```text
isSpoiler == true 이고 revealed == false  → 내용 가림(blinded)
  내용 영역 숨김 + 가림막 안내 표시
  → 사용자가 탭 → AlertDialog 확인 → revealed = true → notifyDataSetChanged()
```

두 값의 차이가 핵심:
- `isSpoiler` : **DB에 저장된 값** (이 리뷰는 스포일러다)
- `revealed` : **화면에서만 쓰는 임시 값** (지금 펼쳐봤다 — DB에 저장 안 함)

→ 그래서 화면을 나갔다 다시 들어오면 다시 가려진 상태가 된다. 또한 `ListView`가 줄을 재활용하므로, `ReviewAdapter`는 else 분기에서 클릭 리스너를 반드시 제거해 엉뚱한 줄에서 다이얼로그가 뜨지 않게 한다.

## 9. 리뷰 작성과 "즉시 반영"되는 평균 별점

관련 파일: `WriteReviewActivity` · `DBHelper.insertReview()` · `DBHelper`(집계) · `Session`

```text
상세 화면 작성(+) 버튼 → WriteReviewActivity (book_id 전달)
  → 별점/한줄평/스포일러 입력 → 저장
  → DBHelper.insertReview() 로 reviews 테이블에 한 줄 추가
       (user_id·nickname은 Session에서)
  → finish() → 상세 화면 복귀
  → BookDetailActivity.onResume()에서 다시 조회 → 평균 별점·리뷰 수 갱신
```

### 평균 별점이 계산되는 원리 (이 앱의 핵심)

별점은 **어디에도 저장하지 않는다.** 조회할 때마다 SQL이 매번 다시 계산한다.

```text
리뷰 수   = base_count + 실제 리뷰 수
별점 합   = base_sum   + 실제 리뷰 별점 합
평균 별점 = 별점 합 / 리뷰 수      (나눗셈은 자바에서, 0 나눗셈 방지)
```

```sql
LEFT JOIN reviews r ON r.book_id = b.book_id
GROUP BY b.book_id
```

- `LEFT JOIN` : 리뷰가 0개인 책도 목록에서 빠지지 않게.
- `COALESCE(SUM,0)` : 리뷰가 없으면 합계가 NULL이 되므로 0으로 방어.
- `GROUP BY b.book_id` : 책별로 묶어서 책마다 따로 평균을 낸다.

→ 별점을 캐시하지 않고 매번 재계산하기 때문에, **내가 리뷰를 쓰면 그 책의 평균 별점과 리뷰 수가 곧바로 바뀐다.**

## 10. 내 리뷰 / 마이페이지 — 전부 Session.userId 기준

관련 파일: `MyReviewActivity` · `MyReviewAdapter` · `MyPageActivity` · `DBHelper.getMyReviews()` · `DBHelper.getMyReviewCount()`

```text
[내 리뷰]   Session.userId() → DBHelper.getMyReviews(uid)
              reviews ⨝ books JOIN → 리뷰 + 책 제목/표지를 한 번에 → ListView
[마이페이지] Session에서 닉네임/아이디 표시
              DBHelper.getMyReviewCount(uid)로 리뷰 수 표시
              로그아웃 → Session.logout()(세션 삭제) → LoginActivity
```

포인트: "내" 데이터는 모두 `Session.userId()`를 조건으로 한 DB 조회다. 로그인한 사람이 바뀌면 보이는 데이터도 자동으로 바뀐다.

## 11. 공통 UI / 보조 도구

### 하단 탭 — `BottomNav` · `include_bottom_nav.xml`
세 탭 화면(Home/MyReview/MyPage)이 같은 XML을 `<include>`하고, `BottomNav.setup()`이 현재 탭 색칠 + 탭 이동을 공통 처리한다. 탭 이동 시 이전 탭은 `finish()`되어 백스택에 탭이 하나만 남는다.

### 표지 이미지 — `Covers` · `ImageLoader`
```text
cover 값이 drawable 이름     → 앱 내부 이미지 리소스 (시드 책)
cover 값이 http... 로 시작    → URL → ImageLoader가 백그라운드 다운로드 → 메인 스레드에서 적용
```
`ImageLoader`는 Glide 없이 직접 만든 로더라, `ListView` 재활용 대비로 ImageView의 tag에 URL을 저장해 두고 다운로드 완료 시 tag가 바뀌었으면 적용하지 않는다(엉뚱한 표지 방지).

### 비동기 처리 — `SearchActivity` · `KakaoApi`
네트워크는 UI 스레드에서 못 하므로 `ExecutorService`로 백그라운드에서 호출하고, 결과는 `Handler(main.post)`로 UI 스레드에 돌아와 목록을 갱신한다.

## 13. 코드를 직접 따라갈 때 추천 순서

흐름을 손으로 짚어보고 싶을 때 이 순서로 열면 막힘이 없다.

```text
1. AndroidManifest.xml   — 시작점이 LoginActivity인 것 확인
2. LoginActivity.java     — 세션 확인 → 홈 이동
3. HomeActivity.java      — 인기 도서 조회, 검색창 TextWatcher, 검색 실행 분기
4. DBHelper.java          — 테이블 3개, getPopularBooks/searchBooks, 평균 별점 집계 ★
5. BookAdapter.java       — Book 데이터를 list_item_book.xml에 연결
6. SearchActivity.java    — 카카오 검색 + insertOrGetBook
7. KakaoApi.java          — API 호출, JSON → Book 파싱
8. BookDetailActivity.java— book_id로 재조회, 정렬, 작성 이동
9. ReviewAdapter.java     — 리뷰 표시, 스포일러 블라인드 ★
10. WriteReviewActivity   — 리뷰 저장 → onResume 재조회로 반영
```

★ 표시 두 곳(DBHelper의 평균 별점, ReviewAdapter의 스포일러)이 이 앱에서 가장 "설계가 들어간" 부분이다.

## 14. Java / Android 최소 개념

흐름을 읽는 데 필요한 단어만.

```text
class            관련 코드 묶음.            예) HomeActivity = 홈 화면 코드 묶음
method           함수.                      예) onCreate(), refreshList(), doSearch()
onCreate()       Activity가 처음 만들어질 때 1번 실행
onResume()       화면이 (다시) 보일 때마다 실행 — 리뷰 저장 후 갱신에 사용
Intent           화면 이동 + 데이터 전달 객체
findViewById()   XML의 버튼/텍스트/리스트를 코드로 가져오기
setOnClickListener()  클릭 시 실행할 코드 등록
Adapter          데이터 리스트 → 화면 목록 아이템으로 변환하는 연결 장치
SQLiteOpenHelper SQLite DB 생성/업그레이드 관리 클래스 (DBHelper의 부모)
SharedPreferences 로그인 상태 같은 간단한 값을 앱 내부에 저장
ExecutorService  네트워크처럼 오래 걸리는 작업을 백그라운드에서 실행
Handler          백그라운드 결과를 다시 UI에 반영할 때 사용
```

## 15. 자주 막히는 지점 정리 (자문자답)

**Q. 별점은 어디에 저장돼 있나?**
저장 안 한다. `books.base_count/base_sum`(기본값)과 `reviews`(실제 리뷰)를 조회할 때마다 SQL이 합쳐서 평균을 다시 계산한다.

**Q. 검색은 DB 검색인가 API 검색인가?**
둘 다. 홈에서 입력 중일 때는 로컬 DB 제목 필터, 키보드 검색 버튼을 누르면 `SearchActivity`에서 카카오 API.

**Q. 리뷰 쓰고 나면 화면이 어떻게 갱신되나?**
저장 후 `finish()`로 상세 화면에 복귀 → `BookDetailActivity.onResume()`이 책/리뷰를 다시 조회 → 평균 별점·리뷰 수 반영.

**Q. 스포일러 "펼침" 상태는 저장되나?**
아니다. `revealed`는 화면에서만 쓰는 임시 값. DB에는 `is_spoiler`(스포일러인지 여부)만 저장된다.

**Q. 같은 상세 화면을 어떻게 여러 경로에서 재사용하나?**
상세 화면은 `book_id`만 받아 DB에서 다시 조회하기 때문에, 진입 경로(홈/검색/내 리뷰)와 무관하게 동작한다.

**Q. 카카오 키가 없으면?**
카카오 검색·표지 보강만 안 되고 안내 문구가 뜬다. 로그인·홈·상세·리뷰 작성 등 로컬 SQLite 기능은 그대로 동작한다.

## 16. 정말 짧은 30초 요약

> BookLog는 Android Java 도서 리뷰 앱이다. 화면은 Activity로 나누고, `SQLiteOpenHelper`인 `DBHelper`가 회원·책·리뷰를 관리한다. 홈은 DB의 책을 리뷰 수(`total_count`) 순으로 보여주고, 검색창 입력은 로컬 제목 필터, 검색 실행은 카카오 API 외부 검색이다. 상세 화면은 `book_id`로 책·리뷰를 다시 조회하며, 평균 별점은 기본값과 실제 리뷰를 합쳐 **조회할 때마다 재계산**하므로 리뷰를 쓰면 즉시 반영된다. "내" 데이터는 모두 `Session.userId()` 기준으로 조회된다.
