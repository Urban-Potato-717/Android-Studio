# BookLog 발표 — 기능별 코드 설명집

PPT(`4조_BookLog_발표PPT`) 슬라이드 순서에 맞춰, 각 기능을 발표할 때 **어떤 코드를 띄우고 무슨 말을 할지** 정리한 치트시트입니다.

- **(보여준다)** = 화면에 띄울 파일/위치
- 그 아래 글 = 실제로 말할 멘트
- `파일:줄번호` 형식으로 위치를 적어둠

## 슬라이드 5 — 강점: SQLite 기반 라이브러리

> PPT 포인트: "회원/도서/리뷰를 서버 없이 앱 내부 SQLite로 관리, 카카오 API로 외부 검색"

이 슬라이드는 앱 전체 구조를 한 번에 말하는 자리라서, **DB가 하나의 중심**이라는 걸 보여주면 됩니다.

**(DBHelper.java:17-33 을 보여준다 — 클래스 선언 + `get()` 싱글톤)**

데이터는 전부 `DBHelper` 한 곳에서 관리합니다. `SQLiteOpenHelper`를 상속받았고, `DBHelper.get(context)`로 앱 전체에서 **단 하나의 DB 객체**만 쓰도록 싱글톤으로 만들었습니다. Room 같은 라이브러리 없이 순수 SQL로 직접 짰습니다.

**(DBHelper.java:36-68 을 보여준다 — `onCreate`의 세 테이블)**

테이블은 세 개입니다. 회원(`users`), 도서(`books`), 리뷰(`reviews`). 슬라이드에 나온 "회원 / 도서 / 리뷰" 세 카드가 정확히 이 세 테이블입니다.

### 회원 — SQLite 계정 로그인 + 비밀번호 보안

**(Pw.java:6-19 를 보여준다)**

비밀번호는 평문으로 저장하지 않습니다. `Pw.hash()`에서 **SHA-256으로 해시**한 뒤 DB에 넣습니다.

**(DBHelper.java:166-184 을 보여준다 — `signup` / `login`)**

회원가입 때도 해시해서 저장하고, 로그인할 때도 입력한 비번을 같은 방식으로 해시해서 **해시끼리 비교**합니다. 그래서 DB가 통째로 노출돼도 원래 비밀번호는 알 수 없습니다.

**(Session.java:20-35 를 보여준다)**

로그인에 성공하면 사용자 정보를 `SharedPreferences`에 저장합니다(`Session.login`). 그래서 앱을 껐다 켜도 로그인이 유지되고, `isLoggedIn()`으로 자동 로그인을 판단합니다.

## 슬라이드 6 — 기능_빠른 적용: 즉시 계산되는 변경사항

> PPT 포인트: "사용자가 작성한 리뷰·별점이 즉시 계산되어 전체 리뷰 수·평균 평점에 바로 반영"

**이 슬라이드가 우리 앱의 핵심 코드입니다.** 평균 별점 계산 방식을 보여주세요.

**(DBHelper.java:189-194 를 보여준다 — `BOOK_AGG_SELECT`)**

핵심은 이 SQL입니다. 책의 평균 별점을 **"미리 넣어둔 커뮤니티 평점 + 실제 사용자 리뷰"를 합쳐서** 계산합니다.

- `total_count` = `base_count`(기본 리뷰 수) + `COUNT(실제 리뷰)`
- `total_sum` = `base_sum`(기본 별점 합) + `SUM(실제 별점)`

`LEFT JOIN`을 쓴 이유는 **리뷰가 0개인 책도 목록에 나오게** 하기 위해서고, `COALESCE(SUM,0)`은 리뷰가 없을 때 합계가 NULL이 되는 걸 0으로 막아줍니다.

**(DBHelper.java:210-211 을 가리킨다 — 평균 계산)**

합계까지는 SQL이 구하고, 나눗셈(평균)은 자바에서 합니다. `평점 합 ÷ 리뷰 수`. 개수가 0이면 0으로 나누지 않도록 막았습니다.

**(WriteReviewActivity.java:57-85 를 보여준다 — 저장 버튼)**

사용자가 별점과 한줄평을 쓰고 저장하면 `insertReview()`로 reviews 테이블에 한 줄이 추가됩니다.

**(BookDetailActivity.java:93-99 를 보여준다 — `onResume`)**

그리고 작성 화면을 닫고 상세 화면으로 돌아오면 `onResume()`에서 `bindBook()`과 `loadReviews()`가 다시 실행됩니다. 별점 계산을 따로 저장해 두는 게 아니라 **화면이 보일 때마다 위의 집계 SQL을 다시 돌리기 때문에**, 내가 쓴 리뷰가 평균 별점과 리뷰 수에 곧바로 반영됩니다. 이게 슬라이드의 "즉시 계산"입니다.

## 슬라이드 7 — 기능_스포일러 방지

> PPT 포인트: "스포일러 포함 리뷰는 원하는 사람만 탭해서 볼 수 있게"

**(WriteReviewActivity.java:45, 79 를 보여준다 — 체크박스 저장)**

먼저 리뷰를 쓸 때 `CheckBox`로 "스포일러 포함" 여부를 받아서, 그 값을 리뷰와 함께 DB에 저장합니다.

**(ReviewAdapter.java:73-83 을 보여준다 — 블라인드 처리)**

목록에서는 두 상태를 구분합니다.
- `isSpoiler` : **DB에 저장된 값** (이 리뷰는 스포일러다)
- `revealed` : **화면에서만 쓰는 임시 값** (지금 펼쳐봤다)

`blinded = isSpoiler && !revealed`, 즉 스포일러인데 아직 안 펼쳤으면 실제 내용을 숨기고(`groupContent` GONE) 가림막 안내를 보여줍니다(`groupSpoiler` VISIBLE).

**(ReviewAdapter.java:81-82 의 `setOnClickListener(null)` 를 강조한다)**

여기가 제일 신경 쓴 부분입니다. `ListView`는 화면 밖으로 넘어간 줄을 **재활용**하기 때문에, 예전에 스포일러였던 줄이 일반 리뷰 자리에 다시 쓰일 수 있습니다. 그때 클릭 리스너를 안 지우면 일반 리뷰인데도 엉뚱하게 다이얼로그가 뜨므로, else 분기에서 **리스너를 반드시 제거**합니다.

**(ReviewAdapter.java:88-100 을 보여준다 — 다이얼로그)**

가림막을 탭하면 `AlertDialog`로 "스포일러 주의" 경고가 뜨고(슬라이드 화면과 동일), 확인을 누르면 그 리뷰의 `revealed`만 true로 바꾼 뒤 `notifyDataSetChanged()`로 다시 그려서 내용을 펼칩니다. `revealed`는 DB에 저장하지 않으므로 **화면을 다시 열면 도로 가려집니다.**

## 슬라이드 8 — 기능_편리한 정렬

> PPT 포인트: "최신순 / 별점순 정렬 선택"

**(BookDetailActivity.java:28-29 를 보여준다 — `sortByRating` 플래그)**

정렬 상태는 `sortByRating` 불리언 하나로 관리합니다. false면 최신순, true면 별점순.

**(BookDetailActivity.java:65-78 을 보여준다 — 정렬 다이얼로그)**

상단의 정렬 글자를 누르면 `AlertDialog`의 `setSingleChoiceItems`(단일 선택 목록)로 기준을 고릅니다. 고르면 `sortByRating` 값을 바꾸고 `loadReviews()`를 다시 호출합니다.

**(DBHelper.java:295-307 을 보여준다 — `getReviewsForBook`)**

정렬은 자바에서 리스트를 직접 정렬하는 게 아니라, **SQL의 `ORDER BY`만 바꿔서 DB가 정렬한 결과를 다시 가져오는** 방식입니다. 별점순이면 `rating DESC`, 최신순이면 `created_at DESC`. 즉 정렬 기준이 바뀌면 쿼리를 다시 돌립니다.

## 슬라이드 9 — 기능_다시 추억하다 (내 리뷰)

> PPT 포인트: "과거 작성한 리뷰를 한 페이지에서 모아 보기"

**(MyReviewActivity.java:46-54 를 보여준다 — `onResume`)**

내 리뷰 탭은 화면이 보일 때마다 현재 로그인한 사용자 id(`Session.userId`)로 그 사람이 쓴 리뷰만 조회합니다.

**(DBHelper.java:309-326 을 보여준다 — `getMyReviews`)**

이 쿼리가 핵심인데, `reviews`와 `books`를 **JOIN**해서 리뷰 내용뿐 아니라 **책 제목과 표지까지 같이** 가져옵니다(`r.user_id = 내 id` 조건). 그래서 리뷰 목록에 책 표지·제목이 함께 보입니다.

**(MyReviewActivity.java:31-32 를 가리킨다 — emptyView)**

리뷰가 하나도 없으면 `setEmptyView`로 "작성한 리뷰 없음" 안내가 대신 뜨도록 처리했습니다.

## 슬라이드 10 — 기능_마이페이지

> PPT 포인트: "내 정보 확인, 작성한 리뷰 수, 로그아웃"

**(MyPageActivity.java:29-31 을 보여준다 — 정보 표시)**

마이페이지는 DB를 다시 조회하지 않고, 로그인할 때 `Session`에 저장해 둔 닉네임·아이디를 그대로 보여줍니다.

**(MyPageActivity.java:44-50 을 보여준다 — 리뷰 수)**

작성한 리뷰 수는 `onResume`에서 `getMyReviewCount()`로 매번 다시 셉니다. 리뷰를 쓰고 돌아오면 숫자가 갱신됩니다. (슬라이드의 "챌린지 = 공유한 리뷰 수")

**(MyPageActivity.java:33-41 을 보여준다 — 로그아웃)**

로그아웃은 `Session.logout()`으로 SharedPreferences의 세션 값을 지우고, 화면 스택을 모두 비운 뒤 로그인 화면으로 보냅니다. 그래서 로그아웃 후 뒤로가기로 홈에 다시 못 들어갑니다.

## 슬라이드 11 — 기타 도구 (카카오 API / 비동기 / 이미지)

> PPT 포인트: "HttpURLConnection 기반 카카오 REST API, ExecutorService/Handler 비동기, URL 이미지 직접 로딩 — 외부 라이브러리 없음"

### 카카오 책 검색 API

**(KakaoApi.java:27-57 을 보여준다)**

외부 도서 검색은 `HttpURLConnection`으로 카카오 REST API를 직접 호출합니다. API 키는 `Authorization` 헤더에 `KakaoAK` 형식으로 넣고, 받은 JSON을 `org.json`으로 파싱해 `Book` 객체로 바꿉니다. Retrofit 같은 라이브러리를 쓰지 않고 표준 자바만으로 구현했습니다.

### 비동기 처리 (네트워크는 UI 스레드에서 못 함)

**(SearchActivity.java:29-31, 94-108 을 보여준다)**

네트워크 요청은 메인 스레드에서 실행할 수 없으므로, `ExecutorService`로 **백그라운드 스레드에서 검색**하고, 결과가 오면 `Handler`(`main.post`)로 **UI 스레드로 돌아와** 목록을 갱신합니다. 이게 안드로이드 비동기의 기본 패턴입니다.

**(SearchActivity.java:50-61 을 보여준다 — `insertOrGetBook`)**

검색 결과는 아직 우리 DB의 책이 아닙니다. 사용자가 결과를 탭해 상세로 들어갈 때만 `insertOrGetBook()`으로 저장하는데, **같은 제목+저자가 이미 있으면 새로 안 넣고 기존 책을 재사용**해서 중복을 막습니다.

### URL 이미지 직접 로딩

**(ImageLoader.java:21-34 을 보여준다)**

카카오가 준 표지 이미지는 URL입니다. Glide·Picasso 같은 라이브러리 없이 `ExecutorService`로 다운로드하고 `Handler`로 화면에 적용합니다. 여기서도 `ListView` 재활용 문제가 있어서, **ImageView의 tag에 현재 URL을 저장**해 두고, 다운로드가 끝났을 때 tag가 바뀌었으면(다른 책으로 재활용됐으면) 적용하지 않습니다(`ImageLoader.java:29`). 엉뚱한 표지가 뜨는 걸 막는 장치입니다.

## 한 줄 마무리 멘트

> "BookLog는 SQLite 하나로 회원·도서·리뷰를 관리하고, 별점은 기본 평점과 실제 리뷰를 합쳐 즉시 계산합니다. 스포일러 블라인드, 정렬, 내 리뷰 모아보기 같은 기능은 모두 수업에서 배운 표준 위젯(ListView·BaseAdapter·AlertDialog·RatingBar)만으로 구현했고, 외부 검색은 카카오 API를 라이브러리 없이 직접 연동했습니다."
